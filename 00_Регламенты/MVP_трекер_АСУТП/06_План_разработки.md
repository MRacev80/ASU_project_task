# MVP-PLAN: план разработки

## Подход

Разработку MVP целесообразно вести короткими итерациями. Сначала собирается локальный контур задач и документов, затем агентские предложения, затем интеграция с LLM API, UI и надежность.

## Этап 0. Подготовка проекта

Статус: готово.

Цель: создать техническую основу приложения.

Задачи:

- выбрать стек реализации;
- создать структуру репозитория;
- настроить локальный запуск приложения;
- настроить backend API;
- настроить серверный UI на HTML-шаблонах;
- подключить SQLite;
- описать локальные настройки.

Результат:

- приложение запускается локально на `localhost`;
- backend отвечает на health-check;
- создана локальная база SQLite.

## Этап 1. Базовая модель данных

Статус: частично готово.

Цель: реализовать хранение основных сущностей MVP.

Задачи:

- создать таблицы `projects`, `tasks`, `documents`, `agents`, `agent_proposals`, `events`, `settings`;
- реализовать миграции базы данных;
- добавить CRUD для проектов, задач и документов;
- писать изменения в журнал событий.

Результат:

- можно создать проект;
- можно создать задачу;
- можно добавить документ как запись с путем к файлу;
- изменения фиксируются в журнале.

## Этап 2. Рабочий экран проекта

Статус: частично готово.

Цель: сделать первый полезный UI без агентной логики.

Задачи:

- реализовать главный экран проекта;
- показать краткую карточку проекта;
- показать список задач;
- добавить фильтры;
- реализовать создание и редактирование задачи;
- реализовать изменение статуса;
- реализовать панель документов;
- реализовать просмотр последних событий.

Результат:

- пользователь может открыть проект и вести задачи вручную;
- пользователь видит документы проекта;
- пользователь видит журнал последних действий.

## Этап 3. Работа с документами

Статус: частично готово.

Цель: подготовить контур документов для агентной обработки.

Задачи:

- добавить документ по локальному пути;
- хранить тип и статус документа;
- связать документ с задачами;
- реализовать резервное копирование;
- реализовать перенос в архив;
- проверять доступность файла;
- добавить расчет хеша файла.

Результат:

- документы управляются из интерфейса;
- документы можно связать с задачами;
- система видит, если документ пропал или изменился.

## Этап 4. Агентские предложения без реального LLM

Статус: не начато.

Цель: реализовать бизнес-поток агентских предложений на тестовых данных.

Задачи:

- создать интерфейс очереди предложений;
- реализовать тестовые предложения;
- реализовать принятие, отклонение и уточнение;
- реализовать преобразование предложения в задачу;
- фиксировать действия в журнале.

Результат:

- проверена схема "агент предлагает, человек подтверждает";
- принятое предложение превращается в задачу.

## Этап 5. Интеграция с LLM API

Цель: подключить реальный агентский анализ ТЗ и устава проекта.

Задачи:

- создать модуль интеграции с облачным LLM API;
- добавить настройки API-ключа и модели;
- реализовать чтение текста из поддерживаемых документов;
- сформировать prompt для анализа;
- получить структурированный список задач, вопросов и рисков;
- сохранить результат как `agent_proposals`;
- обработать ошибки API и недоступность сети.

Результат:

- пользователь запускает анализ ТЗ или устава;
- агент создает предложения;
- предложения не меняют данные без подтверждения.

## Этап 6. Доработка UI и UX

Цель: сделать интерфейс удобным для ежедневной работы.

Задачи:

- привести стиль к минималистичному рабочему виду;
- улучшить таблицу задач;
- добавить счетчики состояния проекта;
- добавить правую панель задачи;
- добавить пустые состояния;
- добавить индикатор выполнения анализа;
- добавить сообщения об ошибках.

Результат:

- интерфейс цельный и спокойный;
- основные сценарии выполняются без лишних переходов.

## Этап 7. Надежность и локальная эксплуатация

Цель: подготовить MVP к реальному локальному использованию.

Задачи:

- добавить резервное копирование базы;
- добавить резервное копирование документов;
- настроить пути к рабочей папке, бэкапу и архиву;
- добавить обработку отсутствующих файлов;
- добавить архивирование вместо удаления;
- подготовить инструкцию по локальному запуску.

Результат:

- систему можно использовать локально;
- есть понятное место хранения данных;
- есть бэкап и архив.

## Этап 8. Проверка первого сценария

Проверяемый поток:

```text
Создать проект -> добавить ТЗ и устав -> запустить агента -> получить предложения -> принять предложения -> получить задачи -> увидеть события в журнале
```

Критерии готовности:

- проект создается;
- документы добавляются;
- агентский анализ запускается;
- предложения появляются в очереди;
- предложения можно принять или отклонить;
- принятые предложения становятся задачами;
- задачи отображаются в списке проекта;
- все ключевые действия отражаются в журнале;
- приложение работает локально на `localhost`.

## Приоритеты реализации

Приоритет 1:

- локальный запуск;
- база данных;
- проекты;
- задачи;
- документы;
- журнал событий.

Приоритет 2:

- предложения агента;
- подтверждение человеком;
- преобразование предложения в задачу.

Приоритет 3:

- LLM API;
- анализ ТЗ и устава;
- структурированное извлечение задач, вопросов и рисков.

Приоритет 4:

- бэкап;
- архив;
- улучшение UI;
- инструкция по эксплуатации.

## Что не делать в первой разработке

- не делать сложную авторизацию;
- не делать полноценную многопользовательскую работу;
- не делать портфельную аналитику;
- не делать сложный календарный план;
- не делать мобильное приложение;
- не делать автоматическое изменение документов агентом;
- не делать экспорт, пока он не потребуется.
