# MVP-ARCH: архитектура и эксплуатация

## Назначение

Документ фиксирует архитектурные решения MVP и требования к локальной эксплуатации.

## Архитектурные решения

| ID | Решение |
|---|---|
| `ADR-001` | Система реализуется как локальное веб-приложение на `localhost` |
| `ADR-002` | Локальная база данных - SQLite |
| `ADR-003` | Документы остаются в текущих папках проекта |
| `ADR-004` | Для документов нужны бэкап и архив |
| `ADR-005` | ИИ-агенты работают через облачные LLM API |
| `ADR-006` | MVP работает без учетных записей пользователей |
| `ADR-007` | Основные документы первого сценария: техническое задание и устав проекта |
| `ADR-008` | Экспорт задач в MVP не требуется |
| `ADR-009` | Все действия агента требуют подтверждения человеком |
| `ADR-012` | Единицей хранения проекта является папка проекта с базой, документами, архивом, бэкапом и агентской рабочей областью |

## Состав приложения

- локальный веб-интерфейс на `localhost`;
- backend API;
- локальная база данных SQLite;
- файловое хранилище документов;
- модуль анализа документов;
- модуль агентов;
- модуль интеграции с облачными LLM API;
- модуль журнала событий;
- модуль резервного копирования и архива документов.

## Поток обработки

```text
Техническое задание / устав проекта -> Анализ агентом -> Предложения -> Проверка человеком -> Задачи проекта
```

## Архитектурные принципы

### Локальность

Система работает локально на машине пользователя. Данные проекта, задачи, журнал и связи с документами хранятся локально.

### Отдельный агентный слой

Агентная логика отделена от бизнес-логики задач. Агент не меняет утвержденные сущности напрямую.

### Событийность и журналирование

Важные действия фиксируются как события. Это нужно для наблюдаемости работы агентов и восстановления истории изменений.

### Человек как точка контроля

В MVP все результаты агентской работы проходят через подтверждение человеком.

### Изолированная интеграция с LLM

Вызовы LLM должны быть изолированы в отдельном модуле, чтобы позже можно было заменить провайдера или добавить локальные модели.

## Локальное хранение

Целевая модель хранения: данные проекта лежат внутри папки проекта.

Рекомендуемая структура:

```text
ProjectFolder/
  project_config.json
  tracker.sqlite
  documents/
  archive/
  backup/
  agents/
  exports/
```

Приложение должно открывать папку проекта, читать `project_config.json` и подключать `tracker.sqlite` внутри этой папки.

## Надежность

Система должна:

- сохранять историю действий;
- минимизировать риск потери данных;
- поддерживать резервное копирование базы;
- поддерживать резервное копирование документов;
- поддерживать архивирование вместо удаления;
- показывать, если документ не найден по сохраненному пути.

## Ограничения первой версии

- работа без учетных записей пользователей;
- ограниченный набор ролей;
- ограниченный набор типов документов;
- агентские действия только через предложения;
- ручное подтверждение всех результатов агентской работы;
- отсутствие сложной интеграции с внешними системами;
- отсутствие полноценной портфельной аналитики;
- отсутствие экспорта задач.

## Минимальная диагностика

В настройках или служебном экране желательно показывать:

- путь к базе;
- путь к папке документов проекта;
- путь к бэкапу;
- путь к архиву;
- доступность LLM API;
- количество задач;
- количество документов;
- количество предложений, ожидающих решения;
- последнюю ошибку агента, если есть.
