# MVP-UI: задание на интерфейс

## Назначение

Интерфейс MVP должен быть простым, минималистичным и рабочим. Основная цель UI - заменить рабочую Excel-карточку проекта единой живой карточкой в браузере: показать состояние проекта, текущие задачи, документы, риски, закупки и предложения ИИ-агентов без визуального шума.

## Общие принципы

- интерфейс похож на рабочий инструмент, а не на презентационный сайт;
- первый экран после запуска - рабочий экран проекта;
- рабочий экран проекта по смыслу соответствует листам `Шаблон проекта.xlsx`;
- крупные hero-блоки, маркетинговые тексты и декоративные элементы не используются;
- дизайн спокойный, светлый или нейтральный;
- основные действия доступны с первого экрана;
- важные состояния видны без перехода по большому количеству страниц;
- интерфейс рассчитан на ноутбук и монитор;
- мобильная версия не является приоритетом MVP.

## Основной layout

Рекомендуемая структура:

1. Верхняя панель.
2. Левая навигационная панель.
3. Центральная рабочая область.
4. Правая вспомогательная панель.

### Верхняя панель

- название системы;
- выбранный проект;
- статус проекта;
- кнопка запуска анализа документа агентом;
- кнопка создания задачи.

### Левая панель

- список проектов или выбранный проект;
- разделы: Задачи, Документы, Предложения агента, Журнал;
- настройки системы.

### Центральная область

- основной список задач;
- фильтры и поиск;
- переключение вида: список / канбан;
- выбранная задача или выбранное предложение.

### Правая панель

- документы проекта;
- предложения агента, ожидающие подтверждения;
- последние события журнала.

## Экран проекта

На экране проекта должны быть:

- название проекта;
- заказчик;
- объект автоматизации;
- текущий статус проекта;
- текущая стадия;
- путь к папке документов;
- краткий блок состояния: всего задач, в работе, заблокировано, ожидают подтверждения;
- список задач;
- список документов;
- очередь предложений агента;
- последние события.

Карточка проекта не должна занимать весь экран. Основное внимание - задачам.

## Вкладки карточки проекта

Рабочий экран проекта должен быть организован как набор вкладок:

- `Главная`;
- `Задачи`;
- `Бюджет`;
- `Закупки`;
- `Документы`;
- `Риски`;
- `P3.express`;
- `Журнал`;
- `Агенты`.

Для первой итерации полноценно реализуются:

- `Главная`;
- `Задачи`;
- `Документы`;
- `Журнал`.

Остальные вкладки могут показывать пустое состояние и краткое описание будущего раздела.

## Список задач

Основной вид задач - таблица или компактный список.

Минимальные колонки:

- статус;
- название;
- тип;
- исполнитель;
- срок;
- приоритет;
- источник;
- признак "требует подтверждения".

Фильтры:

- статус;
- исполнитель;
- тип;
- приоритет;
- источник;
- требует подтверждения;
- заблокированные.

Поиск:

- по названию;
- по описанию;
- по связанному документу.

Строка задачи открывает детальную панель задачи.

## Карточка задачи

Карточка задачи открывается в правой панели или отдельном модальном окне.

Поля:

- название;
- описание;
- тип;
- статус;
- приоритет;
- срок;
- исполнитель;
- проверяющий;
- связанные документы;
- источник;
- результат;
- комментарии;
- история изменений.

Действия:

- сохранить;
- перевести статус;
- заблокировать;
- отправить на проверку;
- принять результат;
- связать документ;
- запустить агента по задаче.

## Экран документов

Поля:

- название;
- тип;
- путь к файлу;
- статус: активный, архивный, отсутствует;
- дата добавления;
- дата последнего анализа;
- связанные задачи.

Действия:

- добавить документ;
- открыть путь к документу;
- запустить анализ агентом;
- отправить документ в архив;
- создать резервную копию.

## Очередь предложений агента

Для каждого предложения показывать:

- тип предложения;
- краткое название;
- содержание;
- источник;
- агент;
- документ-основание;
- дата создания;
- статус рассмотрения.

Действия:

- принять;
- отклонить;
- отправить на уточнение;
- преобразовать в задачу;
- открыть документ-основание.

При подтверждении предложения система явно показывает, какая задача будет создана или какие данные будут изменены.

## Журнал событий

Для каждого события показывать:

- дата и время;
- субъект: человек, агент или система;
- действие;
- объект действия;
- краткое описание.

## Визуальный стиль

- минимальная палитра: нейтральный фон, темный текст, один акцентный цвет;
- статусы задач отличаются цветом, но не слишком яркие;
- элементы управления компактные;
- таблицы и списки хорошо читаемые;
- карточки используются только для отдельных объектов: задача, предложение, документ;
- не использовать крупные декоративные блоки, градиенты, иллюстрации и перегруженные панели.

Цветовая логика:

- Бэклог - серый;
- В работе - синий;
- Тестируется - индиго;
- Готово - зеленый;
- Заблокировано - красный.

## Состояния интерфейса

Необходимо предусмотреть:

- пустой проект без задач;
- проект без документов;
- нет предложений агента;
- агент выполняет анализ;
- ошибка анализа документа;
- документ не найден по сохраненному пути;
- задача заблокирована;
- предложение ожидает подтверждения.

## Минимальный состав экранов

1. Экран проекта.
2. Экран задач.
3. Экран документов.
4. Экран предложений агента.
5. Экран журнала.
6. Экран настроек.

Экраны могут быть реализованы как вкладки внутри одного локального веб-приложения.

## Критерии приемки UI

Пользователь может:

- открыть локальное приложение;
- выбрать или создать проект;
- увидеть список задач проекта;
- создать задачу вручную;
- изменить статус задачи;
- добавить документ проекта;
- запустить анализ технического задания или устава проекта агентом;
- увидеть предложения агента;
- принять или отклонить предложение агента;
- увидеть, что принятое предложение стало задачей;
- открыть журнал действий.
