# MVP-ITER-01: задание на первую итерацию разработки

## Назначение

Первая итерация должна создать технический фундамент MVP: локальное приложение на `localhost`, базовую базу данных SQLite, backend API и минимальный серверный интерфейс для ведения живой карточки проекта вместо Excel-файла.

Цель итерации - получить работающий skeleton, который можно запускать локально и постепенно расширять агентной логикой.

## Границы первой итерации

### Входит

- локальный запуск приложения;
- backend API;
- серверный UI на HTML-шаблонах;
- SQLite база данных;
- базовые таблицы;
- создание и просмотр проектов;
- создание, просмотр и изменение статуса задач;
- добавление документов как записей с локальным путем;
- рабочий экран карточки проекта с вкладками по структуре Excel-шаблона;
- журнал событий для ключевых действий;
- минимальная страница настроек;
- README с инструкцией запуска.

### Не входит

- реальная интеграция с LLM API;
- агентский анализ документов;
- очередь предложений агента;
- бэкап и архив документов;
- авторизация и учетные записи;
- многопользовательская работа;
- экспорт задач;
- портфельная аналитика;
- сложный дизайн UI.

## Предлагаемый стек

Для первой итерации рекомендуется простой монолитный стек:

- приложение: Python + FastAPI;
- UI: Jinja2 templates + HTMX;
- база данных: SQLite;
- ORM: SQLAlchemy;
- миграции: Alembic или простой SQL-инициализатор на старте;
- стили: простой CSS;
- запуск: локально на `localhost`.

React/Vite не используется в первой итерации. Его можно рассмотреть позже, если интерфейс станет сложнее и потребуется отдельное frontend-приложение.

## Минимальные HTTP routes

Первая итерация может совмещать HTML-страницы и JSON API. Для действий из UI желательно предусмотреть HTMX-compatible routes, которые возвращают HTML-фрагменты.

### Служебные

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/health` | Проверка работы backend |

### Страницы

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/` | Главная страница, список проектов или выбранный проект |
| `GET` | `/projects/{project_id}` | Рабочий экран проекта |
| `GET` | `/settings` | Настройки |

### Проекты

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/projects` | Получить список проектов в JSON |
| `POST` | `/projects` | Создать проект из UI |
| `GET` | `/api/projects/{project_id}` | Получить проект в JSON |
| `POST` | `/projects/{project_id}` | Обновить проект из UI |

### Задачи

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/projects/{project_id}/tasks` | Получить задачи проекта в JSON |
| `POST` | `/projects/{project_id}/tasks` | Создать задачу из UI |
| `GET` | `/api/tasks/{task_id}` | Получить задачу в JSON |
| `POST` | `/tasks/{task_id}` | Обновить задачу из UI |
| `POST` | `/tasks/{task_id}/status` | Изменить статус задачи из UI |

### Документы

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/projects/{project_id}/documents` | Получить документы проекта в JSON |
| `POST` | `/projects/{project_id}/documents` | Добавить документ из UI |
| `POST` | `/documents/{document_id}` | Обновить документ из UI |

### Журнал

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/projects/{project_id}/events` | Получить события проекта в JSON |

### Настройки

| Метод | URL | Назначение |
|---|---|---|
| `GET` | `/api/settings` | Получить настройки в JSON |
| `POST` | `/settings` | Обновить настройки из UI |

## Минимальный UI первой итерации

UI должен быть простым и рабочим.

### Экран проекта

На экране:

- список проектов;
- выбранный проект;
- краткая информация о проекте;
- вкладки карточки проекта: Главная, Задачи, Бюджет, Закупки, Документы, Риски, P3.express, Журнал, Агенты;
- список задач;
- список документов;
- последние события.

### Задачи

Пользователь может:

- создать задачу;
- открыть задачу;
- изменить название, описание, тип, приоритет, срок, исполнителя;
- изменить статус задачи;
- увидеть заблокированные задачи.

### Документы

Пользователь может:

- добавить документ как путь к локальному файлу;
- указать тип документа;
- увидеть список документов проекта;
- изменить статус документа.

### Журнал

Пользователь видит последние события:

- создан проект;
- изменен проект;
- создана задача;
- изменена задача;
- изменен статус задачи;
- добавлен документ;
- изменен документ.

## События, которые нужно писать в журнал

- `project.created`;
- `project.updated`;
- `task.created`;
- `task.updated`;
- `task.status_changed`;
- `document.created`;
- `document.updated`;
- `settings.updated`.

## Критерии приемки

Первая итерация считается готовой, если:

- приложение запускается локально;
- backend доступен по `/health`;
- локальный UI открывается в браузере;
- SQLite база создается автоматически или по инструкции;
- можно создать проект;
- можно открыть проект;
- рабочий экран проекта показывает вкладки карточки проекта;
- можно создать задачу в проекте;
- можно изменить статус задачи;
- можно добавить документ с локальным путем;
- действия отображаются в журнале событий;
- после перезапуска приложения данные сохраняются;
- в README описано, как запустить локальное приложение.

## Проверочный сценарий

```text
Запустить приложение -> открыть localhost в браузере -> создать проект -> создать задачу -> перевести задачу в работу -> добавить документ -> открыть журнал -> перезапустить приложение -> убедиться, что данные сохранились
```
