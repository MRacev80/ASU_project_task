# MVP-CONCEPT: концепция, границы и сценарии

## Назначение

Система предназначена для ежедневной работы по задачам внутри конкретного проекта АСУТП. Она должна позволять вести задачи проекта, связывать их с документами, фиксировать действия людей и ИИ-агентов, а также контролировать результаты агентной работы через подтверждение человеком.

Ключевая идея MVP: единое локальное рабочее место, где люди и ИИ-агенты работают в общем контуре задач, документов, решений и проверок.

## Цели MVP

- создать локальный рабочий контур для одного проекта АСУТП;
- вести задачи проекта с понятными статусами, сроками и ответственными;
- подключать документы проекта к задачам;
- позволить агенту анализировать техническое задание и устав проекта;
- обеспечить обязательную проверку и утверждение предложений агента человеком;
- фиксировать историю действий людей и агентов;
- заложить основу для дальнейшего управления портфелем проектов.

## Границы MVP

### Входит

- карточка проекта;
- список задач проекта;
- создание и редактирование задач;
- назначение исполнителя: человек или ИИ-агент;
- статусы задач;
- приоритеты и сроки;
- связь задач с локальными документами;
- загрузка или привязка документа проекта;
- разбор документа агентом;
- очередь предложений агента;
- подтверждение или отклонение предложений человеком;
- комментарии и результаты выполнения задач;
- журнал действий;
- локальное хранение данных;
- резервное копирование документов;
- архив неиспользуемых документов.

### Не входит

- полноценное управление портфелем проектов;
- сложное бюджетирование;
- календарно-сетевое планирование уровня MS Project;
- интеграции с внешними корпоративными системами;
- автоматическое изменение проектной документации без подтверждения;
- сложная ролевая модель;
- облачная синхронизация;
- мобильное приложение;
- учетные записи пользователей и авторизация;
- экспорт задач в первой версии.

## Роли

### Руководитель проекта

- создает и редактирует проект;
- утверждает задачи;
- назначает исполнителей;
- принимает или возвращает результаты;
- просматривает состояние проекта;
- контролирует задачи, созданные агентами.

### Инженер / исполнитель

- просматривает назначенные задачи;
- берет задачу в работу;
- обновляет статус;
- добавляет комментарии;
- прикладывает результат выполнения;
- отправляет задачу на проверку.

### Аналитик / координатор

- добавляет документы проекта;
- запускает разбор документов агентом;
- проверяет предложения агента;
- уточняет формулировки задач;
- следит за полнотой задач и связей с документами.

### ИИ-агент

Агент выполняет вспомогательную интеллектуальную работу. В MVP все его действия требуют подтверждения человеком.

Агент может:

- анализировать документы;
- предлагать задачи;
- предлагать вопросы и риски;
- готовить черновики описаний;
- проверять полноту задачи;
- формировать краткое резюме документа.

Агент не должен:

- самостоятельно утверждать изменения;
- удалять данные без подтверждения;
- менять проектные документы напрямую;
- закрывать задачи без приемки человеком;
- принимать управленческие решения от имени руководителя проекта.

## Статусы задач

Базовый поток:

```text
Бэклог -> В работе -> Тестируется -> Готово
```

Дополнительное состояние:

- Заблокировано.

Для задач, созданных или обработанных агентом, дополнительно фиксируются признаки:

- создано агентом;
- требует проверки человеком;
- принято человеком;
- отклонено человеком.

## Ключевые сценарии

### Сценарий 1. Разбор технического задания и устава проекта

1. Пользователь открывает проект.
2. Пользователь добавляет или выбирает техническое задание и устав проекта.
3. Пользователь запускает разбор документа агентом.
4. Агент анализирует документы.
5. Агент формирует предложения: задачи, вопросы, риски, решения, замечания.
6. Пользователь просматривает предложения.
7. Пользователь принимает, отклоняет или уточняет предложения.
8. Принятые предложения преобразуются в задачи проекта.
9. Все действия фиксируются в журнале.

### Сценарий 2. Ручное создание задачи

1. Пользователь открывает проект.
2. Пользователь создает задачу.
3. Пользователь указывает описание, срок, приоритет и исполнителя.
4. При необходимости пользователь связывает задачу с документом.
5. Задача попадает в рабочий список.
6. Исполнитель берет задачу в работу.
7. После выполнения задача отправляется на проверку.
8. Проверяющий принимает или возвращает задачу на доработку.

### Сценарий 3. Ежедневный контроль состояния проекта

1. Руководитель проекта открывает рабочий экран проекта.
2. Система показывает задачи по статусам.
3. Система выделяет заблокированные задачи.
4. Система показывает задачи, ожидающие проверки.
5. Система показывает предложения агентов, ожидающие решения.
6. Руководитель принимает решения по блокерам и приоритетам.

## Функциональные требования

| ID | Требование |
|---|---|
| `FR-001` | Система должна позволять создать и открыть проект |
| `FR-002` | Система должна позволять создать, редактировать и просматривать задачи |
| `FR-003` | Система должна позволять менять статус задачи |
| `FR-004` | Система должна позволять добавить локальный документ в проект |
| `FR-005` | Система должна позволять связать задачу с документом |
| `FR-006` | Система должна позволять запустить агентский анализ документа |
| `FR-007` | Система должна показывать предложения агента |
| `FR-008` | Человек должен иметь возможность принять или отклонить предложение агента |
| `FR-009` | Принятое предложение должно преобразовываться в задачу |
| `FR-010` | Все значимые действия должны фиксироваться в журнале |

## Нефункциональные требования

| ID | Требование |
|---|---|
| `NFR-001` | Система работает локально на `localhost` |
| `NFR-002` | Данные хранятся на локальной машине |
| `NFR-003` | MVP работает без учетных записей пользователей |
| `NFR-004` | Все действия ИИ-агентов видимы и проверяемы |
| `NFR-005` | Все результаты агентской работы требуют подтверждения человеком |
| `NFR-006` | Система должна поддерживать резервное копирование и архивирование документов |
