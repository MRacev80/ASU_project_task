# MVP-AGENTS: агенты и документы

## Назначение

Документ описывает работу с локальными документами проекта и ИИ-агентами. Главный принцип MVP: агент ничего не меняет напрямую, а создает предложения, которые подтверждает человек.

## Документы проекта

Документы физически остаются в текущих папках проекта. Система хранит путь к файлу, метаданные и связи с задачами.

Типы документов:

- техническое задание;
- устав проекта;
- протокол встречи;
- схема;
- чек-лист;
- регламент;
- письмо;
- отчет;
- прочий документ.

Статусы документов:

- активный;
- архивный;
- отсутствует.

## Основные действия с документами

- добавить документ в проект;
- указать тип документа;
- сохранить локальный путь к файлу;
- связать документ с задачами;
- запустить анализ агентом;
- создать резервную копию;
- перенести документ в архив;
- показать ошибку, если файл отсутствует по сохраненному пути.

## Первый агентский сценарий

```text
Техническое задание и устав проекта -> агент извлекает задачи, вопросы и риски -> человек утверждает -> задачи попадают в проект
```

Этапы:

1. Пользователь выбирает проект.
2. Пользователь добавляет или выбирает техническое задание и устав проекта.
3. Пользователь запускает анализ агентом.
4. Агент читает доступный текст документов.
5. Агент формирует предложения.
6. Система сохраняет предложения в `agent_proposals`.
7. Пользователь принимает, отклоняет или отправляет предложения на уточнение.
8. Принятые предложения преобразуются в задачи.
9. Все действия пишутся в `events`.

## Типы предложений агента

- задача;
- вопрос;
- риск;
- замечание;
- решение;
- изменение.

## Статусы предложений

- новое;
- принято;
- отклонено;
- требует уточнения;
- преобразовано в задачу.

## Правила работы агента

- агент создает только предложения;
- агент не меняет утвержденные данные напрямую;
- агент не удаляет документы и задачи;
- агент не закрывает задачи;
- агент не принимает управленческие решения;
- все действия агента требуют подтверждения человеком;
- все действия агента фиксируются в журнале событий.

## Интеграция с LLM API

В MVP агенты работают через облачные LLM API.

Минимальные требования:

- API-ключ задается в локальных настройках или переменных окружения;
- вызовы LLM изолируются в отдельном модуле;
- ответ агента должен приводиться к структурированному виду;
- ошибки API отображаются пользователю;
- запуск агента, результат и ошибки пишутся в журнал.

## Структура результата агента

Агент должен возвращать структурированный список:

- тип предложения;
- краткое название;
- описание;
- основание в документе;
- предполагаемый приоритет;
- предполагаемый исполнитель, если можно определить;
- комментарий агента;
- уровень уверенности, если поддерживается.

## Требования к подтверждению

Перед принятием предложения пользователь должен видеть:

- что предложил агент;
- из какого документа взято основание;
- какая задача будет создана;
- какие поля будут заполнены;
- что попадет в журнал.

Поддерживаемые действия:

- принять предложение;
- отклонить предложение;
- отправить на уточнение;
- преобразовать в задачу;
- открыть документ-основание.

## Бэкап и архив документов

Для документов необходимо предусмотреть:

- путь к папке резервных копий;
- путь к архиву;
- действие "создать резервную копию";
- действие "перенести в архив";
- статус документа "архивный";
- проверку отсутствующих файлов.
