# MVP-WORKSPACE: папка проекта и работа агентов

## Назначение

Документ фиксирует новую базовую модель хранения и работы: единицей проекта является не отдельный Excel-файл и не глобальная база приложения, а **папка проекта**.

Папка проекта хранится на диске сервера и является единой точкой аккумуляции информации по объекту АСУТП. Приложение открывает папку проекта, читает ее манифест, подключает базу данных проекта и работает с документами внутри этой папки.

Эта модель также задает фундамент для будущей работы ИИ-агентов: агент должен взаимодействовать не с произвольной файловой системой, а с понятным проектным workspace.

## Архитектурное решение

`ADR-012`: единицей хранения проекта является папка проекта. База данных, документы, архив, бэкапы, настройки и агентские артефакты хранятся внутри этой папки. Приложение является инструментом открытия и редактирования такой папки.

## Базовая структура папки проекта

Рекомендуемая структура:

```text
ProjectFolder/
  project_config.json
  tracker.sqlite
  documents/
  archive/
  backup/
  agents/
    inbox/
    work/
    outbox/
    logs/
  exports/
  README.project.md
```

Назначение:

| Путь | Назначение |
|---|---|
| `project_config.json` | Манифест проекта и настройки workspace |
| `tracker.sqlite` | База данных карточки проекта |
| `documents/` | Актуальные документы проекта |
| `archive/` | Архивные или неиспользуемые документы |
| `backup/` | Резервные копии базы и документов |
| `agents/inbox/` | Входные задания и контекст для агентов |
| `agents/work/` | Рабочая область агента для промежуточных файлов |
| `agents/outbox/` | Результаты, предложения и отчеты агентов |
| `agents/logs/` | Логи агентских запусков |
| `exports/` | Будущие выгрузки и отчеты |
| `README.project.md` | Краткое описание проекта для людей и агентов |

## Манифест проекта

Файл `project_config.json` должен быть точкой входа для приложения и агентов.

Минимальная структура:

```json
{
  "project_id": "uuid",
  "project_name": "Название проекта",
  "customer": "Заказчик",
  "automation_object": "Объект автоматизации",
  "database": "tracker.sqlite",
  "documents_dir": "documents",
  "archive_dir": "archive",
  "backup_dir": "backup",
  "agents_dir": "agents",
  "created_at": "2026-05-25T00:00:00",
  "updated_at": "2026-05-25T00:00:00",
  "schema_version": 1
}
```

Правила:

- приложение при открытии папки ищет `project_config.json`;
- если манифеста нет, приложение может предложить создать новый project workspace;
- путь к базе берется из поля `database`;
- относительные пути считаются относительно корня папки проекта;
- изменение манифеста должно фиксироваться в журнале событий.

## Роль базы данных

`tracker.sqlite` хранит структурированные данные карточки проекта:

- проект;
- задачи;
- документы;
- бюджет;
- закупки;
- риски;
- P3.express;
- предложения агентов;
- журнал событий;
- настройки.

Файлы документов физически остаются в папках проекта, а база хранит пути, статусы и связи.

## Роль документов

Папка `documents/` содержит актуальные документы проекта.

Примеры:

```text
documents/
  01_ТЗ/
  02_Устав/
  03_Схемы/
  04_Протоколы/
  05_ПНР/
```

Приложение не обязано навязывать строгую структуру подпапок в MVP, но должно:

- хранить путь к документу;
- уметь показать, что файл отсутствует;
- фиксировать дату последней проверки;
- позволять связать документ с задачей;
- позволять агенту использовать документ как источник анализа.

## Агентская рабочая область

Папка `agents/` нужна, чтобы агентская работа была наблюдаемой и отделенной от утвержденных данных.

### `agents/inbox/`

Сюда помещаются задания для агента и подготовленный контекст.

Примеры:

```text
agents/inbox/
  task_2026-05-25_analyze_tz.json
  context_project_summary.md
```

### `agents/work/`

Временная рабочая зона агента.

Правила:

- агент может создавать здесь промежуточные файлы;
- файлы из `work/` не считаются утвержденными результатами;
- содержимое можно очищать после завершения задачи, если результат сохранен в `outbox/` или базе.

### `agents/outbox/`

Сюда агент складывает результаты.

Примеры:

```text
agents/outbox/
  proposal_tasks_from_tz_2026-05-25.json
  risk_review_2026-05-25.md
  document_summary_2026-05-25.md
```

Результаты из `outbox/` должны импортироваться в приложение как предложения, а не как утвержденные изменения.

### `agents/logs/`

Сюда пишутся технические логи агентских запусков.

Примеры:

```text
agents/logs/
  2026-05-25_120000_analyze_tz.log
```

## Правила работы агента

Агент может:

- читать `project_config.json`;
- читать документы из `documents/`;
- читать утвержденные данные через приложение или экспортированный контекст;
- создавать временные файлы в `agents/work/`;
- сохранять результаты в `agents/outbox/`;
- писать технические логи в `agents/logs/`;
- создавать предложения для задач, рисков, документов и замечаний.

Агент не должен:

- напрямую менять `tracker.sqlite` без слоя приложения;
- удалять документы;
- перезаписывать документы в `documents/`;
- перемещать файлы в `archive/` без подтверждения человека;
- менять `project_config.json` без подтверждения;
- закрывать задачи без приемки человеком;
- считать свои результаты утвержденными.

## Формат задания агенту

Задание агенту можно описывать JSON-файлом в `agents/inbox/`.

Пример:

```json
{
  "task_id": "uuid",
  "type": "analyze_documents",
  "project_id": "uuid",
  "input_documents": [
    "documents/01_ТЗ/ТЗ.docx",
    "documents/02_Устав/Устав.md"
  ],
  "expected_outputs": [
    "task_proposals",
    "risk_proposals",
    "questions"
  ],
  "write_to": "agents/outbox/",
  "requires_human_confirmation": true
}
```

## Формат результата агента

Результат агента должен быть структурированным.

Пример:

```json
{
  "agent": "project_analyst",
  "source_task_id": "uuid",
  "source_documents": [
    "documents/01_ТЗ/ТЗ.docx"
  ],
  "proposals": [
    {
      "type": "task",
      "title": "Проверить перечень сигналов",
      "description": "В ТЗ указан перечень сигналов, требуется сверка с существующими схемами.",
      "source_reference": "ТЗ, раздел 3",
      "priority": "Средний",
      "confidence": 0.78
    }
  ]
}
```

Приложение импортирует такие результаты в таблицу `agent_proposals`.

## Взаимодействие приложения и агента

Рекомендуемый поток:

```text
Пользователь выбирает действие агента
-> приложение создает задание в agents/inbox/
-> агент читает project_config.json и входные документы
-> агент пишет результат в agents/outbox/
-> приложение импортирует результат как предложения
-> человек подтверждает или отклоняет предложения
-> подтвержденные предложения становятся задачами или изменениями
-> все действия пишутся в журнал
```

## Будущий skill для агента

После стабилизации структуры папки проекта можно создать отдельный skill для ИИ-агента.

Skill должен описывать:

- как найти `project_config.json`;
- какие папки можно читать;
- куда писать временные файлы;
- куда писать результаты;
- какие действия запрещены;
- как оформлять предложения;
- как ссылаться на документы-источники;
- как не нарушать правило подтверждения человеком.

До появления skill эти правила должны считаться архитектурным контрактом проекта.

## Изменения для текущего MVP

Текущая реализация пока использует базу:

```text
tracker_mvp/data/tracker.sqlite
```

Следующий архитектурный шаг:

```text
Открыть папку проекта -> найти project_config.json -> подключить tracker.sqlite внутри папки проекта
```

Для плавного перехода можно реализовать:

1. экран выбора или создания папки проекта;
2. создание структуры папки проекта;
3. перенос текущей базы в папку проекта;
4. хранение последней открытой папки в настройках приложения;
5. запрет работы без выбранной папки проекта, кроме режима создания новой папки.
