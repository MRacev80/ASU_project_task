# MVP локального трекера задач АСУТП: индекс и трассировка

## Назначение пакета

Пакет документов описывает MVP локального трекера задач проекта АСУТП с участием ИИ-агентов. Документы разделены так, чтобы разные агенты или исполнители могли работать с отдельными частями без загрузки всего контекста.

## Состав документов

| Код | Документ | Назначение |
|---|---|---|
| `MVP-CONCEPT` | `01_Концепция_и_границы.md` | Цели, границы MVP, роли, сценарии, функциональные и нефункциональные требования |
| `MVP-DATA` | `02_Модель_данных.md` | Сущности, таблицы SQLite, связи, правила целостности |

| `MVP-ARCH` | `05_Архитектура_и_эксплуатация.md` | Архитектурные решения, локальность, хранение, бэкап, ограничения |

## Принятые решения

| ID | Решение | Источник |
|---|---|---|
| `ADR-001` | Система реализуется как локальное веб-приложение на `localhost` | `MVP-ARCH` |
| `ADR-002` | Данные хранятся локально в SQLite | `MVP-DATA`, `MVP-ARCH` |
| `ADR-003` | Документы остаются в текущих папках проекта | `MVP-AGENTS`, `MVP-ARCH` |
| `ADR-004` | Нужны резервное копирование документов и архив неиспользуемых файлов | `MVP-AGENTS`, `MVP-ARCH` |
| `ADR-005` | ИИ-агенты в MVP работают через облачные LLM API | `MVP-AGENTS`, `MVP-ARCH` |
| `ADR-006` | MVP работает без учетных записей пользователей | `MVP-CONCEPT`, `MVP-ARCH` |
| `ADR-007` | Основные документы первого сценария: техническое задание и устав проекта | `MVP-CONCEPT`, `MVP-AGENTS`, `MVP-PLAN` |
| `ADR-008` | Экспорт задач в MVP не требуется | `MVP-CONCEPT`, `MVP-ARCH` |
| `ADR-009` | Статусы задач: Бэклог, В работе, Тестируется, Готово, Заблокировано | `MVP-CONCEPT`, `MVP-DATA`, `MVP-UI` |
| `ADR-010` | Все действия агента требуют подтверждения человеком | `MVP-CONCEPT`, `MVP-AGENTS`, `MVP-ARCH` |
| `ADR-011` | Приложение является живой карточкой проекта и заменяет рабочий Excel-файл как источник истины | `MVP-EXCEL`, `MVP-CONCEPT` |
| `ADR-012` | Единицей хранения проекта является папка проекта с базой, документами, архивом, бэкапом и агентской рабочей областью | `MVP-WORKSPACE`, `MVP-ARCH` |

## Правила поддержки трассировки

- Каждое новое требование получает ID формата `FR-XXX`, `NFR-XXX` или `ADR-XXX`.
- Если требование влияет на данные, UI и архитектуру, оно должно быть отражено во всех соответствующих документах.
- Решения, которые меняют границы MVP, фиксируются в индексе как `ADR`.
- Документы можно отдавать агентам отдельно, но перед объединением результатов нужно сверяться с этой трассировочной таблицей.
