# MVP-EXCEL: привязка Excel-шаблона к приложению

## Назначение

Документ фиксирует, как существующий файл `02_Шаблоны/Шаблон проекта.xlsx` переносится в локальное приложение.

Ключевая идея: приложение должно стать живой карточкой проекта и единой точкой аккумуляции информации по объекту АСУТП. Excel-шаблон используется как исходная модель разделов, полей и типовых списков, но основная работа должна идти в приложении, а не в копиях Excel-файла.

## Проблема текущего процесса

Сейчас карточка проекта ведется как Excel-файл. Это создает проблемы:

- нескольким людям трудно работать одновременно;
- появляются разные версии одного файла;
- изменения приходится пересохранять и объединять вручную;
- при изменении документов проекта нужно повторно фиксировать изменения в карточке;
- нет единого журнала, кто и что изменил;
- сложно понять актуальное состояние проекта без ручной сверки.

## Цель приложения

Приложение должно заменить Excel-карточку проекта как рабочий центр проекта.

Цель:

```text
Один проект -> одна локальная карточка -> одна база данных -> единая актуальная информация
```

Приложение должно аккумулировать:

- общую информацию по проекту;
- состояние проекта;
- контрольные точки;
- задачи;
- бюджет;
- закупки;
- документы;
- риски;
- управленческий чек-лист;
- журнал изменений;
- результаты работы ИИ-агентов.

## Роль Excel-шаблона

Excel-шаблон остается важным артефактом, но меняет роль.

В MVP Excel используется как:

- источник структуры карточки проекта;
- источник типовых задач;
- источник типовых документов;
- источник типовых разделов;
- возможный источник импорта стартовых данных.

Excel не должен оставаться основным местом ежедневной работы.

## Листы Excel и разделы приложения

| Лист Excel | Раздел приложения | Назначение |
|---|---|---|
| `Главная` | `Главная` | Паспорт проекта, состояние, сроки, прогресс, контрольные точки |
| `Задачи` | `Задачи` | Рабочий список задач проекта |
| `Бюджет` | `Бюджет` | План/факт затрат и отклонения |
| `Закупки` | `Закупки` | Позиции закупок, поставщики, статусы |
| `Документы` | `Документы` | Комплект документов проекта и их состояние |
| `Риски` | `Риски` | Реестр рисков и меры снижения |
| `P3.express` | `P3.express` | Управленческий чек-лист проекта |
| `Справка` | `Регламент` | Правила ведения карточки проекта |

Дополнительно в приложении нужен раздел:

| Раздел | Назначение |
|---|---|
| `Журнал` | История изменений людей, системы и агентов |
| `Агенты` | Предложения и результаты работы ИИ-агентов |

## Маппинг данных

### Лист `Главная`

Должен перейти в сущность `project`.

Поля:

- код проекта;
- заказчик;
- объект / название проекта;
- номер договора;
- статус;
- текущая фаза;
- текущая контрольная точка;
- дата начала план;
- дата окончания план;
- дата начала факт;
- дата окончания факт;
- процент готовности;
- стоимость договора;
- бюджет план;
- бюджет факт;
- отклонение;
- руководитель проекта;
- проектировщик;
- программист;
- мастер КИП;
- текущий приоритет / примечание.

Контрольные точки лучше вынести в отдельную таблицу `milestones`.

### Лист `Задачи`

Переходит в сущность `tasks`.

Поля:

- фаза;
- номер задачи;
- задача;
- ответственный;
- дата план;
- дата факт;
- статус;
- комментарий;
- напоминание.

Статусы Excel нужно нормализовать к статусам приложения:

| Excel | Приложение |
|---|---|
| `🔲 В работе` | `В работе` |
| `✅ Готово` | `Готово` |
| `⏸ Ждёт` | `Бэклог` или `Заблокировано` |
| `❌ Отменена` | не входит в базовый поток MVP, можно добавить позже |

### Лист `Бюджет`

Переходит в сущность `budget_items`.

Поля:

- статья затрат;
- план;
- факт;
- отклонение.

Отклонение можно считать автоматически.

### Лист `Закупки`

Переходит в сущность `procurement_items`.

Поля:

- номер;
- наименование;
- количество;
- единица измерения;
- цена план;
- цена факт;
- статус;
- поставщик;
- комментарий.

### Лист `Документы`

Переходит в сущность `documents`.

Поля:

- документ;
- статус;
- дата;
- комментарий;
- локальный путь к файлу;
- дата последней проверки файла;
- признак отсутствующего файла.

В приложении к документам добавляется поле `file_path`, которого в Excel может не быть.

### Лист `Риски`

Переходит в сущность `risks`.

Поля:

- номер;
- описание риска;
- вероятность;
- влияние;
- уровень;
- статус;
- меры по снижению.

Уровень риска можно рассчитывать автоматически.

### Лист `P3.express`

Переходит в сущность `p3_activities`.

Поля:

- номер;
- активность;
- цикл;
- когда;
- ответственный;
- статус;
- дата;
- комментарий.

### Лист `Справка`

Переходит в статический раздел `Регламент` или остается Markdown-документом в папке регламентов.

## Изменения модели данных

К существующим таблицам MVP нужно добавить:

- `milestones`;
- `budget_items`;
- `procurement_items`;
- `risks`;
- `p3_activities`.

Таблицы `projects`, `tasks`, `documents`, `events`, `settings` остаются.

## Изменения UI

Рабочий экран проекта должен быть организован как карточка проекта с вкладками:

1. `Главная`;
2. `Задачи`;
3. `Бюджет`;
4. `Закупки`;
5. `Документы`;
6. `Риски`;
7. `P3.express`;
8. `Журнал`;
9. `Агенты`.

Для первой итерации можно реализовать вкладки визуально, но полноценно наполнить только:

- `Главная`;
- `Задачи`;
- `Документы`;
- `Журнал`.

Остальные вкладки могут показывать пустое состояние и планируемый состав данных.

## Импорт из Excel

Импорт из `Шаблон проекта.xlsx` полезен, но не обязателен для самой первой рабочей версии.

Рекомендуемый порядок:

1. Сначала вручную реализовать структуру приложения.
2. Затем добавить импорт типовых задач, документов, рисков и P3.express из шаблона.
3. Позже добавить экспорт отчета обратно в Excel, если это потребуется пользователям.

## Правила единой точки данных

- Все изменения проекта должны выполняться в приложении.
- Excel-файл не должен редактироваться параллельно как источник истины.
- Документы проекта остаются файлами в папках, но их статус и связи фиксируются в приложении.
- Все изменения должны попадать в журнал событий.
- Если документ изменился или пропал по пути, приложение должно показывать это как состояние документа.

## Решение для совместной работы

Для простоты MVP остается локальным приложением.

Рекомендуемый режим:

- приложение запускается на одной машине в локальной сети;
- участники открывают одну ссылку в браузере;
- данные пишутся в одну SQLite-базу через приложение;
- никто не редактирует копии Excel-карточки параллельно.

Такой режим проще, чем полноценная многопользовательская система, но уже решает проблему множества версий Excel-файла.
